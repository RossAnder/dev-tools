# Plan: Specialised model-pinned sub-agents for flow commands

**Plan path**: `docs/plans/specialised-flow-agents.md`
**Created**: 2026-04-29
**Status**: Draft

## Context

A token-efficiency audit of `~/.claude/commands/` (10 flow commands totalling ~5,872 lines) found that ~14 research-style sub-agent spawns per typical flow cycle inherit the parent session's model — usually Opus 4.7 — when Sonnet handles the fetch-and-summarise workload fine; verification spawns (run tests/build, report pass/fail) similarly inherit Opus when Haiku suffices. The audit also surfaced a drift hazard: ~40 lines of orphan-surfacing and deferred-item-reopen-sweep prose duplicate between `review.md` and `optimise.md` outside the parity manifest, so the pre-commit hook can't catch divergence.

This plan addresses both findings in one cohesive change:

1. Add four selectively-applied specialised agent definitions (`flow-research`, `flow-implement-lite`, `flow-implement-deep`, `verification`) with model pins that match the workload — Sonnet for fetch-and-summarise research, Opus for judgement-heavy implementation, Sonnet for clearly-mechanical apply work, Haiku for pass/fail verification.
2. Migrate 9 flow commands to dispatch the new agents on non-judgement workloads (orchestrator-side selectivity rules; agents do not self-escalate).
3. Reconcile the prose drift between `review.md` and `optimise.md`, then promote the consolidated content to a 4th shared block (`ledger-disposition-sweep`) in the parity manifest.
4. Pilot end-to-end on `/optimise` + `/optimise-apply` (covers research migration AND lite/deep escalation) before broadening to the remaining 7 commands.

### Drift between `review.md` and `optimise.md` orphan-surfacing prose

The candidate content (orphan-surfacing + deferred-item reopen sweep + clock-skew validation) is **NOT byte-identical**. Five distinct semantic divergences:

1. Step references — `review.md` says "before Step 2" / "before scoping the agent launch"; `optimise.md` says "before Step 1.5".
2. Item-ID prefixes in examples — `R7` / `R12` / `R{n}` (review) vs `O7` / `O12` / `O{n}` (optimise).
3. Command-name redirects — "run `/review`" (review.md) vs "run `/optimise`" (optimise.md).
4. `optimise.md` carries an additional fallback clause (~18 words) for an older `tomlctl` binary that doesn't appear in `review.md`.
5. Ledger-prefix references in the deferred-item prompts (R{n} vs O{n} again).

**Reconciliation strategy** (per User Decision §1): replace the divergent fragments with byte-identical placeholders before promotion — generic ID prefix `<id>{n}` (e.g. `<id>7`, `<id>12`); generic command-redirect text "run the active flow command"; generic step reference "before the dispatch section"; lift the optimise-only `tomlctl` fallback clause OUTSIDE the soon-to-be-shared block. After reconciliation, content is byte-identical and the manifest entry can be added safely.

## Approach

### The four agents

Defaults are inverted from the original audit brief in light of User Decision §3: **`-deep` is the default for cluster/per-task apply work; `-lite` is the opt-in path when lite-eligibility holds**.

#### `flow-research` (Sonnet)
- **Workload**: Context7 + WebSearch fetch-and-summarise. Output is structured findings with hard caps (≤500 words / ≤10 findings) against a fixed record template. No deep judgement — synthesis happens in the orchestrator.
- **Tools**: `Glob, Grep, Read, WebSearch, WebFetch, mcp__plugin_context7_context7__query-docs, mcp__plugin_context7_context7__resolve-library-id` (NO `Edit`, `Write`, `Bash`).
- **System prompt** absorbs: Context7-first / WebSearch-second contract; mandatory finding-record template (`Library/API`, `Source`, `Finding`, `Details`, `Impact`); ≤500-word / ≤10-finding default caps with truncation priority (API signatures + version-specific behaviour over narrative); version-pinning requirement; Context7 no-match / multi-match handling.

#### `flow-implement-lite` (Sonnet)
- **Workload**: Mechanical, fully-specified code application where the orchestrator pre-analysed the hard cases.
- **Tools**: `Read, Edit, Write, Glob, Grep, Bash, mcp__plugin_context7_context7__query-docs, mcp__plugin_context7_context7__resolve-library-id`.
- **System prompt** absorbs: applied/skipped tag form (`applied O{n}: ...` / `skipped O{n}: ...`); Tier-2 already-applied protocol; no-overlapping-edits rule; plan-deviation reporting protocol; commit-discipline rules (no force-push, no `--no-verify`).

#### `flow-implement-deep` (Opus)
- **Workload**: Default for apply/implement work. Cross-file refactors, ambiguous-spec arbitration, security-sensitive code paths, anything requiring judgement.
- **Tools**: identical to `-lite`.
- **System prompt**: identical mechanics as `-lite` PLUS guidance for cross-file dependency reasoning, ambiguous-spec arbitration (when finding describes the symptom but not the precise fix), security-sensitive code-path checks (auth, crypto, input validation, sandbox boundaries), and explicit "if unsure, halt and surface to orchestrator rather than guess" escalation policy.

#### `verification` (Haiku)
- **Workload**: Run a build/test/lint command, report `pass`|`fail` with the exact command. No aggregation, no chain-of-thought.
- **Tools**: `Bash, Read, Grep`.
- **System prompt** (tiny): "Run the command provided in the prompt. Return: (a) the exact command run, (b) `pass` or `fail`, (c) the last 20 lines of output if `fail`. Do not interpret. Do not aggregate. Do not retry."

### Lite-eligibility gate (the orchestrator-side decision)

For `/optimise-apply` Step 4, `/review-apply` Step 4, and `/implement` Phase 2 batches, the orchestrator (running on Opus) evaluates each **cluster as a whole** (not individual items within it) against ALL of the following criteria. Dispatch to `flow-implement-lite` ONLY when ALL hold for EVERY item in the cluster; otherwise dispatch the entire cluster to `flow-implement-deep`.

1. **File scope**: the cluster as a whole touches ≤ 2 files, OR for `/implement` the plan task is tagged effort=S.
2. **Action fully specified**: every item's `summary` + `details` (or for `/implement`, the task's `Action` + `Detail` fields) describe the exact change to make. No design decisions are left to the implementer for ANY item in the cluster.
3. **No cross-file refactor**: no item in the cluster requires coordinated edits to call sites, type definitions, or interfaces in files outside the cluster.
4. **Not security-sensitive**: no item in the cluster touches auth, crypto, input-validation, sandbox-boundary, or token-storage code.

**Cluster boundaries are not re-drawn for cost savings.** Cluster boundaries are determined by the existing dependency-sort + file-clustering protocol (per the `apply-dependency-sort` shared block). The orchestrator does NOT peel trivial items out of a cluster into their own batches in order to dispatch them on `-lite` while their cluster-mates run on `-deep` — clean cluster isolation outweighs the marginal cost saving. Stated as a hard rule:

> **Coupling-isolation rule**: if any item in a cluster fails any of the four criteria, the entire cluster goes to `-deep`. A trivial item that is dependency-linked or file-overlapping with a complex item rides with that complex item to `-deep`. Only standalone clusters whose every item passes every criterion go to `-lite`.

The same rule applies to `/implement`: if a plan batch contains a mix of S/M/L tasks with file-overlap or `Depends on:` edges between them, the whole batch goes to `-deep`. Lite is reserved for batches whose every task is independently lite-eligible.

The orchestrator MUST record its lite-vs-deep choice in the agent's prompt header (one line, e.g. `DISPATCH: flow-implement-lite — cluster passes lite-eligibility (1 file, fully-specified action, no cross-cutting impact, non-security path, no coupled deep items)`) so the decision is auditable from the execution record. If any criterion fails — including the coupling-isolation rule — escalate to `-deep` and record the failing criterion explicitly (e.g. `DISPATCH: flow-implement-deep — coupling-isolation: cluster contains item O5 (severity=critical, category=memory) which fails criterion #4`).

This gate is **separate from** the existing critical-finding user-confirmation gate at `optimise-apply.md:549-550` (severity=critical AND category∈{memory,query,concurrency}), which remains in place — that gate suppresses silent automated wontapply transitions, not lite/deep selection.

## Risks

- **Risk**: Reconciling prose drift in Task 5 produces a normalised text that reads worse than the originals (over-genericised) — readers lose context-specific signposting.
  **Mitigation**: Keep substitutions surgical: only the 5 specific divergences identified in Research Notes get touched; surrounding paragraphs stay verbatim. Re-read both carriers post-edit and confirm the prose still flows. If quality regresses, fall back to "promote only the truly-byte-identical sub-block" (User Decision §1 option B).

- **Risk**: The orchestrator's lite-eligibility classification is unreliable in practice — too many false positives (lite dispatched for work that needed deep) or false negatives (deep dispatched for trivial work, no cost saving).
  **Mitigation**: Mandate the per-dispatch decision-rationale recording in the prompt header so post-pilot review can audit accuracy. If the pilot reveals systematic misclassification, tighten criterion #2 (action fully specified) — currently the most subjective gate. The four criteria are AND-combined; the bias is conservative (toward deep), which is the intended failure mode (Opus is a no-quality-loss fallback).

- **Risk**: Sonnet's `flow-research` agent produces lower-quality findings than Opus did, missing subtle issues.
  **Mitigation**: The pilot (`/optimise tomlctl` end-to-end) is the early-warning signal. If finding quality regresses materially, downgrade `flow-research` to `model: opus` (one frontmatter edit) and accept the smaller cost-saving from prompt-shrinkage alone.

- **Risk**: Haiku's `verification` agent fails to surface a subtle test failure (truncates the wrong 20 lines, mis-labels a flaky pass as pass).
  **Mitigation**: Verification is a binary pass/fail decision over a deterministic command; little judgement involved. If false-negatives appear, the agent's system prompt can require the full output (no truncation) — Haiku's context window handles it fine.

- **Risk**: Agent files sync from `~/.claude/agents/` is per-machine; rolling these out to other dev machines requires the user to recreate the four files there.
  **Mitigation**: Out of scope for this plan, but flag in the plan-completion summary so the user knows the agent files are user-global, not repo-tracked. Consider a follow-up plan to track the agent files in dev-tools and add a sync step.

- **Risk**: Existing `general-purpose` boilerplate stripped from commands during migration removes load-bearing instructions that the agent's system prompt does not actually cover.
  **Mitigation**: Strip ONLY the boilerplate explicitly absorbed by each agent's system prompt (Context7 contract / finding-record template / caps for `flow-research`; tag form / no-overlapping-edits / Tier-2 protocol for the `flow-implement-*` agents). Per-call instructions (lens focus, file lists, scope filters, ≤N-word caps) stay verbatim. Do a side-by-side diff per migrated command before committing.

- **Risk**: Post-approval bootstrap of the flow's `.claude/flows/<slug>/` directory and `execution-record.toml` (per `/plan-new` Phase 7) cannot happen during plan mode — those Write operations are forbidden by the harness.
  **Mitigation**: Note this explicitly in the post-approval handoff: after `ExitPlanMode`, the user (or a fresh `/plan-new` continuation) materialises the flow registration steps. The plan content lives at `docs/plans/specialised-flow-agents.md` for now; the canonical `docs/plans/<slug>.md` location and flow-directory bootstrap follow approval.
